# Architecture Guidelines

Overview of the multi-stage Docker build architecture, Caddy serving configuration, and how consumer apps integrate with this shared infrastructure.

## System Overview

```text
Consumer frontend app (e.g., insights-chrome, notifications-frontend)
│
├── src/                     ← App source code
├── package.json             ← Must have insights.appname
├── package-lock.json        ← Or yarn.lock
├── LICENSE                  ← Required by Dockerfile
└── build-tools/             ← Git submodule → this repo
    ├── Dockerfile           ← Multi-stage build definition
    ├── universal_build.sh   ← Build orchestrator
    └── ...

Build command: podman build -f build-tools/Dockerfile .

Final image: Caddy UBI serving static files on port 8000
```

## Multi-Stage Build Architecture

### Stage 1: Builder (Node.js)

Base image: `registry.access.redhat.com/ubi9/nodejs-22`

Purpose: Install dependencies, build the frontend app, generate metadata.

```text
/opt/app-root/
├── bin/
│   ├── universal_build.sh    ← Copied from build-tools/
│   ├── build_app_info.sh     ← Copied from build-tools/
│   └── server_config_gen.sh  ← Copied from build-tools/
└── src/
    ├── package.json          ← App source (COPY . .)
    ├── node_modules/         ← Installed by npm ci / yarn install
    ├── dist/                 ← Build output
    ├── Caddyfile             ← Generated by server_config_gen.sh
    └── app.info.json         ← Generated by build_app_info.sh
```

Key operations:
1. `parse-secrets.sh` loads Konflux secrets
2. Auto-detect Sentry token from `{APP_NAME}_SECRET`
3. Detect npm vs yarn from lock files
4. `npm ci` / `yarn install --immutable`
5. `npm run build` / `yarn build:prod` (or custom script)
6. `build_app_info.sh` generates build metadata JSON
7. `server_config_gen.sh` generates Caddy config

### Stage 2: Runtime (Caddy)

Base image: `quay.io/redhat-services-prod/hcm-eng-prod-tenant/caddy-ubi:latest`

Purpose: Serve the built static files with Caddy web server.

```text
/
├── etc/caddy/Caddyfile       ← From builder stage
├── licenses/LICENSE          ← Required for compliance
└── srv/
    ├── dist/                 ← Build output from builder
    │   ├── index.html
    │   ├── css/
    │   ├── js/
    │   ├── app.info.json     ← Build metadata
    │   └── ...
    └── package.json          ← For runtime metadata
```

### Hermetic Build Variant

`Dockerfile.hermetic` differs from the standard build:

| Aspect | Standard | Hermetic |
|--------|----------|----------|
| Base (runtime) | Caddy UBI | ubi-micro (minimal) |
| Dependencies | Online npm ci | Offline npm ci (Cachi2 prefetch) |
| Serving | Caddy on port 8000 | No server (static files only) |
| Output path | `/srv/dist` | `/srv/dist` |
| Cypress | Not handled | `CYPRESS_INSTALL_BINARY=0` |
| Red Hat labels | Not included | Included for compliance |

## Caddy Configuration

Generated Caddyfile structure:

```text
Port 8000: Application traffic
  ├── /apps/{APP_NAME}/*    → serves from /srv/dist
  ├── {ENV_PUBLIC_PATH}/*   → env-based route, serves from /srv/dist
  └── /                     → redirects to /apps/chrome/index.html

Port 9000: Metrics
  └── /metrics              → Prometheus metrics endpoint
```

### Route Priority

1. **App match**: `/apps/{APP_NAME}/*` — strips prefix, serves static files
2. **Env match**: `{ENV_PUBLIC_PATH}/*` — env-configurable route prefix
3. **Root redirect**: `/` → `/apps/chrome/index.html` (permanent)

### TLS Configuration

- `CADDY_TLS_MODE` controls TLS behavior (default: `http_port 8000`)
- `CADDY_TLS_CERT` placeholder for certificate configuration
- `auto_https disable_redirects` prevents automatic HTTPS redirects

## Build Metadata (app.info.json)

Generated by `build_app_info.sh`, contains:

```json
{
  "app_name": "my-app",
  "node_version": ">=18",
  "src_hash": "abc123...",
  "src_tag": "v1.2.0",
  "src_branch": "main",
  "app_version": "1.2.0",
  "patternfly_dependencies": ["@patternfly/react-core@5.4.0"],
  "rh_cloud_services_dependencies": ["@redhat-cloud-services/frontend-components@4.2.0"]
}
```

This file is served at `/apps/{APP_NAME}/app.info.json` and used for deployment verification and debugging.

## Consumer Integration

### Adding This Repo as a Submodule

```bash
git submodule add https://github.com/RedHatInsights/insights-frontend-builder-common.git build-tools
```

### Required Consumer Files

| File | Requirement |
|------|-------------|
| `package.json` | Must have `insights.appname` field |
| `package-lock.json` or `yarn.lock` | Exactly one must exist |
| `LICENSE` | Copied to `/licenses/` for compliance |
| Build output | Must produce a `dist/` directory (or custom `APP_BUILD_DIR`) |

### Consumer package.json Requirements

```json
{
  "insights": {
    "appname": "my-app-name"
  },
  "scripts": {
    "build": "webpack --config webpack.config.js"
  }
}
```

The `insights.appname` is critical:
- Used for Caddy route generation (`/apps/{appname}/`)
- Used for Sentry token lookup (`{APPNAME}_SECRET`)
- Included in `app.info.json`

### Tekton/Konflux Pipeline Configuration

Standard build in `.tekton/` pipeline YAML:

```yaml
spec:
  params:
  - name: dockerfile
    value: /build-tools/Dockerfile
```

Hermetic build adds prefetch:

```yaml
spec:
  params:
  - name: dockerfile
    value: /build-tools/Dockerfile.hermetic
  - name: prefetch-input
    value: '{"type": "npm", "path": "."}'
```

## Adding New Capabilities

### New Build Argument

1. Add `ARG` + `ENV` in `Dockerfile` (builder stage, with comment block)
2. If needed at runtime, also add in stage 2
3. Use in `universal_build.sh` or relevant script
4. Add test in `test/test_dockerfile_env_vars.py`
5. Document in `AGENTS.md` "Key Build Arguments" table

### New Build Script

1. Create the script in repo root (not in `src/`)
2. Add `set -euo pipefail` header
3. Add `COPY` in Dockerfile builder stage to `/opt/app-root/bin/`
4. Call from `universal_build.sh` or Dockerfile `RUN`
5. Add to `AGENTS.md` project structure and documentation

### New Test Suite

1. Create `test/test_dockerfile_<domain>.py`
2. Add marker in `conftest.py`
3. Add Makefile target
4. Add CI workflow step with 15-minute timeout
5. Follow existing class pattern (see [testing-guidelines.md](testing-guidelines.md))

### Modifying Caddy Configuration

1. Edit `server_config_gen.sh` `generate_caddy_config()` function
2. Test with `make test-caddy`
3. Verify routes work with trailing and without trailing slashes
4. Ensure metrics endpoint on port 9000 is preserved
5. Check that `ENV_PUBLIC_PATH` env-based routing still works
